## TL;DR

- Les **MCP** (Model Context Protocol) sont la façon canonique de connecter Claude Code à des outils et données externes. Le standard open-source est passé en 2026 d'expérimental à dominant.
- Référence agrégée : [`punkpeye/awesome-mcp-servers`](https://github.com/punkpeye/awesome-mcp-servers) (86k★) et [`modelcontextprotocol/servers`](https://github.com/modelcontextprotocol/servers) (85k★, repo officiel).
- 6 catégories couvrent 80 % des besoins : **dev (GitHub, Chrome DevTools, Playwright, Serena)**, **data (Supabase)**, **design (shadcn, Figma)**, **navigation web (Context7)**, **automation (n8n, activepieces)**, **frameworks builders (FastMCP)**.
- Conseil : commencer par 3 serveurs ciblés sur ton stack, pas 30 d'un coup. Les MCP consomment du contexte et peuvent ralentir Claude.

<Callout type="info" title="Qu'est-ce qu'un MCP au juste">
Un MCP server expose un ensemble d'outils (fonctions) et de ressources (données) que Claude peut appeler. Le protocole est standardisé : tout serveur conforme marche avec Claude Code, Cursor, Codex, Gemini CLI, etc. Voir [`/mcp/what-are-mcps`](/mcp/what-are-mcps) pour le détail.
</Callout>

## Comment choisir

Trois principes :

1. **Ne pas tout brancher** : chaque MCP actif consomme du contexte au démarrage. Limite-toi à 5-7 serveurs en simultané, sinon Claude rame.
2. **Profils par contexte** : utilise les **profils MCP** pour avoir un setup "dev" différent du setup "audit SEO" ou "audit sécurité" (cf. [`/advanced/mcp-profiles`](/advanced/mcp-profiles)).
3. **Faire confiance à l'officiel** : pour la production, préférer les MCP des éditeurs (GitHub, Microsoft, Supabase, Anthropic). Les MCP community sont parfaits pour expérimenter.

<Callout type="warning" title="Sécurité MCP">
Un MCP peut accéder à des ressources sensibles (filesystem, base de données, navigateur). Avant d'installer un MCP non officiel, audit du code source obligatoire. Ne jamais lancer un MCP demandant des tokens API en dev local sans vérifier ce qu'il en fait.
</Callout>
